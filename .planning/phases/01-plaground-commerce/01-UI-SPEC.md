## Spacing Scale

Declared values (all multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge internal padding, inline tight gaps |
| sm | 8px | Compact element spacing, input internal padding, chip gaps |
| md | 16px | Default element spacing, card internal padding |
| lg | 24px | Section padding, form group gaps |
| xl | 32px | Layout column gaps, content section breaks |
| 2xl | 48px | Major section breaks (storefront), admin queue row groups |
| 3xl | 64px | Page-level hero/section spacing (storefront only) |

Exceptions:
- Touch targets (mobile): minimum 44px height on all interactive elements (WCAG 2.5.5)
- Admin table rows: 40px row height minimum for keyboard navigability
- Storefront max content width: 1280px; hero sections may break wider (full bleed)
- Admin layout: 12-column grid with 24px gutters at desktop breakpoint

## Typography

### Storefront + Customer Portal

4 sizes, 2 weights. Subsection hierarchy is expressed through weight contrast (700 vs 400), not
additional size steps.

| Role | Font | Size | Weight | Line Height |
|------|------|------|--------|-------------|
| Display (hero, brand moments) | Coolvetica | 48px | 700 | 1.1 |
| Heading h1 (section headings, h2 subsections) | Inter | 32px | 700 | 1.2 |
| Body | Inter | 16px | 400 | 1.5 |
| Label / meta / badge | Inter | 14px | 400 | 1.4 |

**Subsection differentiation note:** h2-level subsections use 32px at weight 700 identically to
h1. Visual hierarchy between h1 and h2 is achieved through spacing context (larger top margin on
h1), not a separate font size. This keeps the scale at exactly 4 declared sizes.

### Admin Dashboard

2 sizes, 2 weights. Emphasis via weight and spacing, not size jumps.

| Role | Font | Size | Weight | Line Height |
|------|------|------|--------|-------------|
| Page heading | Inter | 20px | 600 | 1.3 |
| Section heading / table header | Inter | 14px | 600 | 1.4 |
| Body / table cell | Inter | 14px | 400 | 1.5 |
| Label / metadata / badge | Inter | 12px | 400 | 1.4 |

**Admin typography rationale:** Tighter scale than storefront; emphasis via weight and spacing
rather than size jumps. Strong numeric glyph rendering at 14px for dashboard KPI values is
mandatory — Inter satisfies this requirement.

## Component Design Rules

### Border Radius

| Context | Radius |
|---------|--------|
| Default (buttons, inputs, cards) | 10px |
| Chips, badges, status pills | 6px |
| Hero surfaces, upload drop zone | 16px |
| Tooltips, dropdown menus | 8px |

### Buttons

| Variant | Storefront | Admin |
|---------|-----------|-------|
| Primary | `bg-[#B81D20] text-white` | `bg-[#005EB0] text-white` |
| Secondary | `border border-zinc-300 bg-transparent` | `border border-zinc-700 bg-transparent` |
| Destructive | `bg-[#EF4444] text-white` | `bg-[#EF4444] text-white` |
| Ghost | `text-zinc-900 hover:bg-zinc-100` | `text-zinc-300 hover:bg-zinc-800` |

- Primary buttons do not change semantic meaning based on surface context.
- Destructive buttons are red on both surfaces. No exceptions.
- All buttons must have a visible focus ring (2px solid brand accent color, 2px offset).

### Inputs

- Always visible label above the input — no placeholder-as-label pattern.
- Inline validation: error message appears below the input field, never as toast-only.
- Focus ring: 2px solid brand accent, 2px offset.
- Error state: red border + error message below, icon optional.

### Motion

| Context | Duration |
|---------|----------|
| Micro (button press, chip toggle) | 120–150ms |
| Panel transitions (drawer, sheet, dialog) | 180–220ms |
| Page route transitions | 200–260ms |
| Upload progress, quote compute | Indeterminate with labeled progress bar |

- All motion uses Framer Motion 11.x.
- Skeletons must match the final layout geometry before content loads.
- Chart animations trigger only on first mount or filter change — never loop.
- Tooltips are crisp; crosshair/hover states subtle.
- Charts use neutral tones + one accent series highlight only (PLA Blue for admin, PLA Red for storefront).
- Respect `prefers-reduced-motion` — disable all non-essential animations when set.

### Charts (Admin KPI Dashboard)

- Library: Recharts (wrapped in `packages/ui` as `<KpiLineChart>`, `<KpiBarChart>`, `<KpiDonutChart>`).
- Chart wrapper components are in `packages/ui/` — neither `apps/web-admin` nor `apps/web-storefront` import Recharts directly.
- Color series: use neutral zinc tones for secondary series; PLA Blue (`#005EB0`) for the primary/key series.
- Grid lines: subtle zinc-800 (`#27272A`) horizontal lines only; no vertical grid lines.
- Tooltip background: `#1A1A1A`; border `#2A2A2A`; text `#F4F4F5`.
- Animate on first mount only. No looping.
- Every chart must have a text-based fallback accessible title (`aria-label`).
- Crosshair on hover: thin 1px zinc-600 line; dot indicator on active series point.

## Layout Contracts

### Storefront

- Max content width: 1280px, centered with `mx-auto px-4 sm:px-6 lg:px-8`.
- Hero sections: full-bleed background, constrained content within 1280px.
- Product grid: 1 col mobile → 2 col tablet → 3–4 col desktop.
- Sticky filter sidebar on desktop PLP; bottom-sheet filters on mobile.
- Checkout: single-column on mobile, two-column (form + summary) at ≥768px.
- Mobile-first: primary CTA visible above fold on all key screens.

### Storefront — Product Card

- One strong image (aspect ratio 4:3, object-fit cover).
- Tight product title (max 2 lines, truncate).
- Short spec line (material, color — max 1 line, muted).
- Price in accent color; lead time below price in muted text.
- Optional material badge: small, muted chip in top-left corner.
- Hover reveals "Quick add" and "View details" actions — no modal overlay.

### Storefront — Product Detail Page (PDP)

- Image gallery first (left column ≥768px).
- Stable buy box (right column): options → material/color first, then size/variant, then quantity.
- Trust blocks below buy box: shipping, QA, support, returns.

### Admin Dashboard

- Left rail navigation: 240px wide; collapses to icon-only (64px) at screens <1280px.
- Nav sections: Orders, Quotes, Products, Customers, Inventory, Settings.
- Content area: 12-column grid, 24px gutters.
- Queue tables: full width with sticky headers, row density toggle.
- KPI cards: 4-column grid at ≥1280px, 2-column at ≥768px, 1-column on mobile.
- Detail drawers: 480px wide, slide in from right, overlay on mobile.
- Quick actions in table rows aligned to the right column.
- Dashboard home uses a modular card/panel system: KPI, queue, alert, and activity widgets share one
  card language and can be rearranged in an explicit "Edit layout" mode.
- Interaction reference: aim for the same clean, high-signal, movable-panel feeling as modern
  printer dashboards such as Bambuddy, while keeping PLAground's own color system, typography, and
  component shapes. Do not directly copy another product's layout, copy, or visual assets.
- Reordering behavior: drag handles are visible only in layout-edit mode; widgets snap to the
  underlying grid, animate to new positions in <220ms, and persist per-admin preferences.
- Recovery path: admin dashboard includes a "Reset layout" action that restores the default widget
  arrangement without affecting underlying data or filters.

### Admin — Operational Clarity Rule

Every admin screen must visually answer three questions in order:
1. **"What needs attention right now?"** — queue depth, expiring auth windows, manual review flags are prominently surfaced at the top.
2. **"What changed?"** — audit trail and status timeline visible without navigating away.
3. **"What action is safe to take next?"** — the primary action button is always the blue accent; destructive actions are always secondary/ghost until confirmed.

## Stripe Auth Window Pattern

The Stripe payment authorization capture window is 5 days (Visa, post-April 2024). The admin queue must surface expiring authorizations before they lapse.

| Time to expiry | Visual treatment |
|----------------|-----------------|
| > 72 hours | No indicator |
| 48–72 hours | Yellow chip "Auth expires in Xd" next to order row |
| < 48 hours | Yellow chip + row highlighted with zinc-800 left border accent |
| Expired | Red chip "Auth expired" + order blocked from capture; action: "Re-authorize" |

This warning pattern uses the yellow (`#FBC70E`) semantic slot — never a new color.

## Four-State Contract

Every surface that loads or mutates data must implement all four states. This is a hard requirement
from CLAUDE.md conventions.

### Loading

| Surface | Skeleton Strategy |
|---------|-----------------|
| Product grid | Card skeletons matching image + title + price geometry |
| Quote result | Skeleton matching line-item breakdown layout |
| Admin queue table | Row skeletons with correct column widths |
| KPI dashboard | Widget skeletons at correct card dimensions |
| File upload | Progress bar with labeled stage ("Uploading...", "Scanning...", "Analyzing...") |
| Chart widgets | Skeleton at chart card dimensions; no spinner |

### Empty

| Surface | Heading | Body | CTA |
|---------|---------|------|-----|
| Product catalog | "Nothing here yet." | "This category is still being stocked. Check back soon or explore other categories." | "Browse all products" |
| Customer order history | "No orders yet." | "Your completed orders will appear here." | "Shop now" |
| Customer quote history | "No quotes yet." | "Upload a 3D model to get your first instant estimate." | "Get a quote" |
| Admin order queue | "Queue is clear." | "No orders need attention right now." | None |
| Admin quote queue | "No quotes pending." | "Submitted quotes will appear here for review." | None |
| Admin product list | "No products yet." | "Add your first product to start selling." | "Add product" |

### Success

- Checkout completion: confirmation page with order number, summary, and "Download receipt" link.
- Quote ready: cost breakdown card with "Instant Estimate" badge and "Order this quote" CTA.
- Quote converted to order: portal order entry appears with status timeline starting at "Placed".
- Admin approval: row status chip updates inline; toast notification "Quote approved".
- File upload accepted: progress completes, transitions to "Analyzing model…" stage.

### Error

| Trigger | Heading | Body | Action |
|---------|---------|------|--------|
| Payment declined | "Payment not processed." | "Your card was declined. Please check your details or try a different payment method." | "Update payment" |
| File type rejected | "File not supported." | "We accept STL and 3MF files. OBJ and STEP files are accepted for manual review." | "Choose a different file" |
| File too large | "File is too large." | "Model files must be under [limit]. Try reducing polygon count before uploading." | "Choose a different file" |
| Quote compute error | "Couldn't estimate this model." | "Something went wrong analyzing your file. Our team has been notified. Try again or contact us." | "Try again" + "Contact us" |
| Generic API error | "Something went wrong." | "We hit an unexpected error. If this keeps happening, contact support." | "Try again" + "Copy error details" (admin/portal only) |
| Session expired | "You've been signed out." | "Your session expired. Sign in again to continue." | "Sign in" |

## Accessibility Contract

- Color contrast minimum: 4.5:1 for normal text, 3:1 for large text and UI components (WCAG AA).
- Focus rings: visible on all interactive elements, 2px solid brand accent, 2px offset.
- All form inputs have associated `<label>` elements — no aria-label substitutes unless input is
  genuinely icon-only.
- Status chips always use both color and label text — never color alone.
- Keyboard-navigable: all interactive flows completable without a pointer device.
- `aria-live` regions on quote progress, file upload progress, and async form submission feedback.
- Reduced motion: all Framer Motion animations respect `prefers-reduced-motion`.
- Charts: every chart has an `aria-label` describing the data represented.
