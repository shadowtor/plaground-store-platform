## Spacing Scale

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge internal padding, inline tight gaps |
| sm | 8px | Compact element spacing, input internal padding, chip gaps |

| 2xl | 48px | Major section breaks (storefront), admin queue row groups |
| 3xl | 64px | Page-level hero/section spacing (storefront only) |

Exceptions:
- Touch targets (mobile): minimum 44px height on all interactive elements (WCAG 2.5.5)
- Admin table rows: 40px row height minimum for keyboard navigability
- Storefront max content width: 1280px; hero sections may break wider (full bleed)
- Admin layout: 12-column grid with 24px gutters at desktop breakpoint

## Component Design Rules

### Border Radius

| Context | Radius |
|---------|--------|
| Default (buttons, inputs, cards) | 10px |
| Chips, badges, status pills | 6px |
| Hero surfaces, upload drop zone | 16px |
| Tooltips, dropdown menus | 8px |

### Buttons

| Variant | Storefront | Admin |
|---------|-----------|-------|
| Primary | `bg-[#B81D20] text-white` | `bg-[#005EB0] text-white` |
| Secondary | `border border-zinc-300 bg-transparent` | `border border-zinc-700 bg-transparent` |
| Destructive | `bg-[#EF4444] text-white` | `bg-[#EF4444] text-white` |
| Ghost | `text-zinc-900 hover:bg-zinc-100` | `text-zinc-300 hover:bg-zinc-800` |

- Primary buttons do not change semantic meaning based on surface context.
- Destructive buttons are red on both surfaces. No exceptions.
- All buttons must have a visible focus ring (2px offset, brand accent color).

### Inputs

- Always visible label above the input â€” no placeholder-as-label pattern.
- Inline validation: error message appears below the input field, never as toast-only.
- Focus ring: 2px solid brand accent, 2px offset.
- Error state: red border + error message below, icon optional.

### Motion

| Context | Duration |
|---------|----------|
| Micro (button press, chip toggle) | 120â€“150ms |
| Panel transitions (drawer, sheet, dialog) | 180â€“220ms |
| Page route transitions | 200â€“260ms |
| Upload progress, quote compute | Indeterminate with labeled progress bar |

- All motion uses Framer Motion.
- Skeletons must match the final layout geometry before content loads.
- Chart animations trigger only on first mount or filter change â€” never loop.
- Respect `prefers-reduced-motion` â€” disable non-essential animations.

## Four-State Contract

Every surface that loads or mutates data must implement all four states. This is a hard requirement
from CLAUDE.md.

### Loading

| Surface | Skeleton Strategy |
|---------|-----------------|
| Product grid | Card skeletons matching image + title + price geometry |
| Quote result | Skeleton matching line-item breakdown layout |
| Admin queue table | Row skeletons with correct column widths |
| KPI dashboard | Widget skeletons at correct card dimensions |
| File upload | Progress bar with labeled stage ("Uploading...", "Scanning...", "Analyzing...") |

### Empty

| Surface | Heading | Body | CTA |
|---------|---------|------|-----|
| Product catalog | "Nothing here yet." | "This category is still being stocked. Check back soon or explore other categories." | "Browse all products" |
| Customer order history | "No orders yet." | "Your completed orders will appear here." | "Shop now" |
| Customer quote history | "No quotes yet." | "Upload a 3D model to get your first instant estimate." | "Get a quote" |
| Admin order queue | "Queue is clear." | "No orders need attention right now." | None |
| Admin quote queue | "No quotes pending." | "Submitted quotes will appear here for review." | None |
| Admin product list | "No products yet." | "Add your first product to start selling." | "Add product" |

### Success

- Checkout completion: confirmation page with order number, summary, and "Download receipt" link.
- Quote ready: cost breakdown card with "Instant Estimate" badge and "Order this quote" CTA.
- Quote converted to order: portal order entry appears with status timeline starting at "Placed".
- Admin approval: row status chip updates inline; toast notification "Quote approved".
- File upload accepted: progress completes, transitions to "Analyzing modelâ€¦" stage.

### Error

| Trigger | Heading | Body | Action |
|---------|---------|------|--------|
| Payment declined | "Payment not processed." | "Your card was declined. Please check your details or try a different payment method." | "Update payment" |
| File type rejected | "File not supported." | "We accept STL and 3MF files. OBJ and STEP files are accepted for manual review." | "Choose a different file" |
| File too large | "File is too large." | "Model files must be under [limit]. Try reducing polygon count before uploading." | "Choose a different file" |
| Quote compute error | "Couldn't estimate this model." | "Something went wrong analyzing your file. Our team has been notified. Try again or contact us." | "Try again" + "Contact us" |
| Generic API error | "Something went wrong." | "We hit an unexpected error. If this keeps happening, contact support." | "Try again" + "Copy error details" (admin/portal only) |
| Session expired | "You've been signed out." | "Your session expired. Sign in again to continue." | "Sign in" |

## Accessibility Contract

- Color contrast minimum: 4.5:1 for normal text, 3:1 for large text and UI components (WCAG AA).
- Focus rings: visible on all interactive elements, 2px solid, 2px offset.
- All form inputs have associated `<label>` elements â€” no aria-label substitutes unless input is
  genuinely icon-only.
- Status chips always use both color and label text â€” never color alone.
- Keyboard-navigable: all interactive flows completable without a pointer device.
- `aria-live` regions on quote progress, file upload progress, and async form submission feedback.
- Reduced motion: all Framer Motion animations respect `prefers-reduced-motion`.
